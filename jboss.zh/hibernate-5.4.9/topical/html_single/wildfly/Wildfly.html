在WildFly中使用最新的Hibernate ORM

WildFly中的HibernateORM

WildFly应用程序服务器开箱即可包含Hibernate ORM作为默认的JPA提供程序。

这意味着您不需要将Hibernate ORM与在WildFly上部署的应用程序打包在一起，相反，如果应用程序服务器检测到您的应用程序正在使用JPA，它将自动启用Hibernate支持。

在不使用JPA时，您还可以从这些模块中受益，从而避免将Hibernate ORM及其所有依赖项包括到您的部署中。这将需要使用jboss-deployment-structure.xml文件或清单条目：有关某些示例，请参见WildFly中的类加载 。

通常，比给定的WildFly版本附带的Hibernate ORM的更新版本要多。为确保您可以在我们发布的WildFly功能包的任何合理的WildFly版本中享受最新版本的Hibernate，这些功能包可以与各种WildFly设置工具一起使用，以使用其他版本的Hibernate ORM创建自定义服务器。

什么是WildFly功能包

WildFly是基于JBoss模块构建的运行时；这是一种轻便高效的模块化类加载器，可将现代服务器的不同组件定义为独立的模块。

Hibernate ORM及其组件被定义为一个这样的模块。这意味着您甚至可以在同一运行时中拥有多个不同版本的Hibernate ORM模块，同时使它们的类路径彼此隔离：您可以将最新的Hibernate ORM版本添加到WildFly，而不必删除现有副本。

这使您可以灵活地为一个应用程序使用最新版本，而不必担心会破坏需要其他版本Hibernate的其他应用程序。通常，我们不建议您滥用此系统，但是例如一次能够升级和测试一个应用程序通常很有用，而不是必须为多个服务强制使用新版本，而必须一次完成所有更新。

功能部件包是一个zip文件，其中包含一些XML文件，这些文件定义了JBoss模块的结构，并列出了Java“ jar”文件，通过Maven坐标来识别它们将需要这些文件。

这还有其他好处：

  • 功能部件包非常小，因为它只是带有几行XML的压缩文件。

  • 就磁盘空间而言，您可以构建一个“瘦”服务器，该服务器实际上不包含您的Maven工件的副本，而只是根据需要从本地Maven缓存中加载类。

  • 您仍然可以选择构建“完整”服务器，以便可以在不复制本地Maven存储库的情况下对其进行重新分发。

  • 使用配置工具时，您将从一种可组合的方法中受益，因此可以将N个不同的包组合在一起以形成一个定制服务器。

  • 由于功能部件包XML仅列出了推荐的工件（而不是包括二进制副本），因此可以轻松覆盖确切的版本。这是应用微型紧急修复程序的理想选择。

  • 一个功能部件包可以声明对其他功能部件包的传递依赖关系，因此将自动为您提供Hibernate ORM的所有非可选依赖关系。

突出它不是什么也很有趣：与大多数构建系统不同，JBoss Modules的重点不是构建项目的方式，而是项目的运行方式。

一个重要方面是JBoss模块的运行时依赖性不是可传递的 ：例如，如果最新的Hibernate ORM需要Byte Buddy版本5（例如），而您的应用程序需要的任何其他模块都取决于Byte Buddy版本6，则不会是一个问题。

升级应用程序并不容易，因为您不必确保所有依赖项都一致以使用相同版本。

如何获取WildFly的最新Hibernate ORM功能包

可以从Maven Central下载该功能包，以方便在构建过程中自动解包。每当发布任何新版本的Hibernate ORM时，都会发布这样的功能包。

范例1。WildFly功能包的Maven标识符
   <groupId>org.hibernate</groupId>
   <artifactId>hibernate-orm-jbossmodules</artifactId>
   <version>5.4.9.Final</version>

通常，您不会直接下载此文件，但会使用Maven插件或Gradle插件来构建自定义WildFly服务器。

创建配置文件

您将需要一个小的XML文件来定义要组装的功能部件包。

以下示例将创建完整的WildFly服务器，但还将包括最新的Hibernate ORM模块的副本：

示例2配给配置文件示例
<server-provisioning xmlns="urn:wildfly:server-provisioning:1.1" copy-module-artifacts="true">
	<feature-packs>
		<feature-pack
				groupId="org.hibernate"
				artifactId="hibernate-orm-jbossmodules"
				version="${hibernate-orm.version}" />
		<feature-pack
				groupId="org.wildfly"
				artifactId="wildfly-feature-pack"
				version="${wildfly.version}" />
	</feature-packs>
</server-provisioning>

当然，如果您希望自定义服务器具有更多功能，则可以列出其他功能包。

也可以通过不设置copy-module-artifacts标志来构建“瘦”服务器，或者可以进一步自定义和过滤要删除的内容。

有关更多详细信息，请参见Github 上的WildFly Build Tools项目的自述文件 。

接下来，您可以使用Maven插件或Gradle插件实际创建自定义服务器的新副本。

Maven用户：调用WildFly Provisioning插件

假设先前的配置文件另存为server-provisioning.xml ，您只需要引用插件，选择输出目录名称，并将插件绑定到构建生命周期。

范例3。示例Maven置备
<build>
	<plugins>
		<plugin>
			<groupId>org.wildfly.build</groupId>
			<artifactId>wildfly-server-provisioning-maven-plugin</artifactId>
			<executions>
				<execution>
				<id>server-provisioning</id>
				<goals>
					<goal>build</goal>
				</goals>
				<phase>compile</phase>
				<configuration>
					<config-file>server-provisioning.xml</config-file>
					<server-name>wildfly-custom</server-name>
				</configuration>
			</execution>

JPA版本覆盖

考虑到使用JavaEE7构建WildFly 12，它附带了JPA 2.1 API。

Hibernate ORM 5.3要求使用JPA 2.2，目前无法使用Maven设置插件替换JPA API，因此您还必须应用“ WildFly补丁”。

可以从WildFly CLI应用WildFly补丁。在这里，我们展示了如何使用Maven插件将其自动化。

示例4用于在WildFly中修补JPA版本的示例Maven脚本：
<plugin>
	<artifactId>maven-dependency-plugin</artifactId>
	<executions>
		<execution>
			<id>fetch-jpa-patch</id>
			<phase>process-test-resources</phase>
			<goals>
				<goal>copy</goal>
			</goals>
			<configuration>
				<artifactItems>
					<artifactItem>
					   <groupId>org.hibernate.javax.persistence</groupId>
					   <artifactId>hibernate-jpa-api-2.2-wildflymodules</artifactId>
					   <classifier>wildfly-12.0.0.Final-patch</classifier>
					   <version>1.0.0.Beta2</version>
					   <type>zip</type>
					   <outputDirectory>${project.build.directory}</outputDirectory>
					   <overWrite>true</overWrite>
					   <destFileName>hibernate-jpa-api-2.2-wildflymodules-patch.zip</destFileName>
					</artifactItem>
				</artifactItems>
			</configuration>
		</execution>
	</executions>
</plugin>
<plugin>
	<groupId>org.wildfly.plugins</groupId>
	<artifactId>wildfly-maven-plugin</artifactId>
	<executions>
		<execution>
			<id>apply-wildfly-jpa22-patch-file</id>
			<phase>pre-integration-test</phase>
			<goals>
				<goal>execute-commands</goal>
			</goals>
			<configuration>
				<offline>true</offline>
				<jbossHome>${jbossHome.provisionedPath}</jbossHome>
				<!-- The CLI script below will fail if the patch was already applied in a previous build -->
				<fail-on-error>false</fail-on-error>
				<commands>
					<command>patch apply --override-all ${project.build.directory}/hibernate-jpa-api-2.2-wildflymodules-patch.zip</command>
				</commands>
			</configuration>
		</execution>
	</executions>
</plugin>

Gradle用户：调用WildFly Provisioning插件

也可以使用Gradle插件，在这种情况下，只需几行。

请记住，在创建“瘦服务器”时：WildFly类加载器将无法从本地Gradle缓存中加载jar：这可能会触发第二次下载，因为它专门查看本地Maven存储库。特别是如果您正在使用Gradle开发其他功能包，请确保将它们发布到Maven存储库中，以便WildFly可以加载它们。

遵循完整的Gradle构建脚本；与前面的Maven示例（为了使它简短而不完整）相反，它是一个可以正常运行的构建脚本。另外，它不需要应用其他补丁来替换JPA版本。

示例5示例Gradle置备
plugins {
  id "org.wildfly.build.provision" version '0.0.6'
}

repositories {
	mavenLocal()
	mavenCentral()
	maven {
		name 'jboss-public'
		url 'https://repository.jboss.org/nexus/content/groups/public/'
	}
}

provision {
	//Optional destination directory:
	destinationDir = file("wildfly-custom")

	//Update the JPA API:
	override( 'org.hibernate.javax.persistence:hibernate-jpa-2.1-api' ) {
		groupId = 'javax.persistence'
		artifactId = 'javax.persistence-api'
		version = '2.2'
	}
	configuration = file( 'wildfly-server-provisioning.xml' )
	//Define variables which need replacing in the provisioning configuration!
	variables['wildfly.version'] = '12.0.0.Final'
	variables['hibernate-orm.version'] = '5.3.0.Final'
}

您可以将其粘贴到名为build.gradle在一个空目录中，然后调用：

gradle provision

您将拥有完整的WildFly 12.0.0。在中生成的最终服务器wildfly-custom子目录，包括Hibernate ORM版本5.3.0的副本。最终版本（除了WildFly通常包含的任何其他版本）。

关于存储库的注释：

mavenLocal（）

完全没有必要，但是如果您不止一次运行它，将使您的构建更快。

老板

此附加存储库是必需的。Maven Central提供了WildFly的大多数组件，但偶尔也有例外。

JPA版本覆盖

JPA API是应用服务器的基本组件，因为它用于与各种其他标准集成。在此阶段，虽然功能部件包提供了一定程度的可组合性，但尚不可能拥有JPA API的其他独立副本：需要对其进行替换。

Hibernate ORM 5.3.0需要JPA 2.2，而WildFly 12附带了JPA 2.1版。幸运的是，该供应工具还能够覆盖任何工件解析。

当然，当将来的WildFly版本将基于JPA 2.2时，可能很快将不再需要此步骤。

WildFly模块标识符：插槽和约定

请注意，WildFly附带的Hibernate ORM模块将保持不变：您可以根据需要从ZIP文件在原始版本和新版本之间进行切换。不同的应用程序可以使用不同的版本。

应用服务器使用名称和插槽标识模块。默认情况下，模块org.hibernate：main将用于为给定的部署提供JPA支持： main是默认插槽，并表示WildFly自身随附的Hibernate ORM副本。

按照惯例，WildFly随附的所有模块都使用“主”插槽，而Hibernate项目发布的模块将使用与该版本匹配的插槽名称，并且还提供一个别名来与其“ major.minor”版本匹配。

我们的建议是使用“ major.minor”表示形式依赖该模块，因为这简化了Hibernate ORM的错误修正版本（微观版本更新）的发布，而无需更改应用程序配置（总是期望微观版本向后兼容并作为错误修正发布。只要）。

例如，如果您的应用程序想要使用最新版本的Hibernate ORM版本5.4.x，则应声明使用模块org.hibernate：5.4 。如果应用程序需要非常特定的版本，您当然可以决定使用完整版本来进行更精确的控制。

切换到其他HibernateORM插槽

为了使用默认org.hibernate：main以外的其他模块，请通过来指定要使用的模块的标识符。 jboss.as.jpa.providerModule应用程序的persistence.xml文件中的属性，如以下示例所示。

范例6。使用Hibernate ORM的备用模块插槽
<persistence xmlns="http://xmlns.jcp.org/xml/ns/persistence"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://xmlns.jcp.org/xml/ns/persistence
    http://xmlns.jcp.org/xml/ns/persistence/persistence_2_1.xsd"
    version="2.1" >

    <persistence-unit name="examplePu">

        <!-- ... -->

        <properties>
            <property name="jboss.as.jpa.providerModule" value="org.hibernate:5.4"/>
        </properties>

        <!-- ... -->

    </persistence-unit>
</persistence>

不用说，这将影响应用程序的类路径：如果单个应用程序声明了多个持久性单元，则它们都应做出一致的选择！

《 WildFly JPA参考指南》中记录了此属性；您可能要检查一下它，因为它列出了其他几个有用的属性。

使用自定义WildFly模块的局限性

使用功能部件包提供的自定义模块时，您将放弃应用程序服务器通常会自动进行的某些集成。

例如，当使用默认的Hibernate ORM模块时，启用Infinispan二级缓存很简单，因为WildFly将自动设置对Infinispan和群集组件的依赖关系。使用这些自定义模块时，这种集成将不再自动进行：您仍可以启用所有通常可用的功能，但是这些功能需要显式配置，就像在其他容器中或没有容器中运行Hibernate一样。

您也许可以从Infinispan或Ehcache项目中获得匹配的功能部件包，也可以自己创建一个模块（毕竟这只是一个简单的XML文件），也可以像以前一样在应用程序中添加其他依赖项：模块和功能部件包可以为您带来一些优势，但是好的老式jar仍然是可行的选择。

不用说，那些对拥有最新版本不感兴趣的用户可以使用WildFly中集成的版本，并受益于WildFly团队精心测试的库组合。
